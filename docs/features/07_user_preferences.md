# 用户偏好卡：证据驱动分析内核

## 目标与边界

偏好卡要回答“玩家希望获得什么体验”，而不是重复“玩家做过什么”。它是可修正的软先验，不是要求正文每回合迎合的硬规则，也不是对真实玩家作心理诊断。

- 正文结算只记录中性的行为证据，不直接推断心理原因，因此不新增每回合模型请求。
- 深度分析累积到一定证据量后低频异步运行；失败不阻塞正文。
- AI 只提出多个可竞争、可并存的解释，并为证据支持或反对某个假设的强度选词。
- Python 将 `weak / moderate / strong` 映射成固定似然比并严格更新每个独立假设的后验。
- 候选不进入正文；推断项只有后验至少 `0.8` 且证据跨至少两个故事才自动生效。
- 玩家明确声明作为 `0.75` 的高先验直接生效，但仍可修正、停用或删除。
- 完整行为证据不会因为一次分析已运行就删除；手动重新分析可让近期旧证据支持之前漏掉的新解释。

## 两阶段协议

### 阶段一：同回合记录事实

正文结算可以附带：

```json
{
  "preference_evidence": [{
    "signal_type": "choice|reroll|retry|undo|generation|declaration|other",
    "summary": "只描述玩家做了什么",
    "context": "理解行为所需的最少情境",
    "diagnosticity": "weak|moderate|strong",
    "sensitive": false
  }]
}
```

普通角色台词、角色心理、战术选择、求胜行为或一次随机结果不利不能直接变成偏好。没有分析价值时返回空数组。

### 阶段二：低频深度分析

偏好分析模型读取尚未分析的证据和已有假设，返回多个假设与逐条证据评估：

- `direction`: `support / against / neutral`
- `strength`: `weak / moderate / strong`
- `coverage_note`: 本轮解释无法区分或没有覆盖的原因
- `missing_possibilities`: 当前尚未想到、需要更多证据或玩家说明的可能性

模型不得输出概率。Python 使用固定似然比：支持为 `1.25 / 2 / 4`，反对为 `0.8 / 0.5 / 0.25`，中性为 `1`，通过赔率形式计算后验。多个原因不是互斥分类，不要求总和为 100%。

这不能保证 AI 第一次就想到真实原因，但能避免“遗漏的解释被当成不存在”：原始证据永久保留；覆盖缺口对玩家可见；手动重新分析允许近期旧证据建立新假设；玩家修正始终高于自动推断。

## 玩家交互

- 正文输入支持纯命令 `/偏好 我喜欢……`；它只更新偏好卡，不推进故事。
- 混合输入支持 `[[偏好：……]]`；保存偏好后，只把去掉元信息的玩家行动发给正文。
- 普通角色扮演语句不会因为包含“喜欢”而被当成玩家声明。
- 顶部偏好入口在出现待确认候选时显示红点，不弹窗打断正文。
- 偏好卡允许选择剧情发展、色情内容、动作描写、角色偏好、关系互动、视觉等分类，并可分别在故事设置中开关。
- 敏感证据分析默认关闭；可配置独立偏好分析 API，留空则继承正文模型。
- 偏好注入要求自然变化地出现，并考虑近期已经满足的内容，不能机械复读。
- 偏好页提供最近证据时间线，明确区分“正文记录”和“界面行为”，并可展开查看每个候选采用的证据方向、强度与 Python 后验依据。

### 独立交互证据

撤回、重试、重掷未来、主动生图和重新生图由 Python 在真实接口成功后记录，不依赖正文模型记得报告。它们统一是弱证据：

- 撤回只能说明玩家不接受该回合整体结果，不能确定原因。
- 重试可能源于文风、内容、角色表现或单纯输出质量。
- 重掷可能源于情节偏好，也可能只是追求有利结果。
- 生图只说明玩家此时希望视觉化；重新生图可能源于画质、构图、一致性或内容。

所有界面交互证据都不复制玩家行动、故事正文、图片或生图提示词，只保存操作类型和关联回合号，避免在未明确标记敏感性的情况下把成人内容发给偏好分析模型。故事结算证据随撤回删除；已经真实发生的界面交互证据不会因为故事时间回溯而消失。

## 数据与回溯

全局本地文件为 `data/preferences/user_preferences.yml`，属于个人资产，不进入 Git。行为证据记录来源故事和回合。撤回某回合时，删除该回合贡献的行为证据与评估，并由 Python 从剩余评估重新计算后验；其他故事的证据不受影响。

## 当前尚未完成

- 对重试、重掷、撤回和生图行为作更细的原因反馈；当前只记录保守弱证据，不让玩家每次填写原因。
- 基于“近期已满足次数”的确定性冷却或物理随机频率控制；当前先使用提示词软约束。
- 多用户、多角色身份和按世界隔离的偏好作用域。
- 完整对话式偏好工坊，以及角色/世界书/文风/实体共用的顶层工坊工作区。
- 语义去重、假设合并与长期证据归档。

顶层“游戏 / 聊天 / 工坊”不必重写整个前端，但也不只是换提示词：它需要抽离工作区路由、左栏上下文模型、右栏资产装载语义和会话持久化。当前先稳定偏好领域内核；至少两个资产工坊成熟后，再把弹窗工坊迁移成共享工作区，避免先造空壳。
