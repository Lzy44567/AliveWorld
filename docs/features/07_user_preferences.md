# 用户偏好卡：渐进学习 MVP

## 目标

用户偏好卡不是要求玩家长期手填的另一张角色卡。它应在正常游玩中逐渐总结玩家对叙事、角色、关系、内容、视觉与边界的长期偏好，并允许玩家低成本纠错。

首版必须满足：

- 不新增每回合独立 LLM 请求；复用正文结算 JSON 返回偏好观察。
- 角色扮演行为不等于真实玩家偏好。
- 单次含糊行为只进入候选，不立刻控制正文。
- 玩家明确表达且证据可信时可立即激活；非明确偏好通常需要重复证据。
- 候选、已生效、已停用状态可检查、修正、停用和删除。
- 停止学习时，不再拼接偏好学习提示词；停止使用时，不向正文注入偏好卡。
- 偏好文件属于本地个人数据，不进入 Git。

## 当前实现

正文结算可附带 `preference_observations`：

```json
{
  "target_id": "可为空；强化已有偏好时填写",
  "category": "narrative|character|relationship|visual|content|boundary|other",
  "polarity": "prefer|avoid",
  "statement": "可长期复用的简洁偏好",
  "evidence": "本回合玩家话语中的依据",
  "confidence": 0.82,
  "explicit": true
}
```

后端保存证据来源、存档名、回合 ID、玩家原始行动摘要、可信度和累计次数。默认规则：

- `explicit=true` 且可信度至少 `0.72`：直接进入已生效。
- 非明确观察：首次进入候选；累计两次且可信度至少 `0.65` 后进入已生效。
- 可信度低于 `0.55` 或缺少依据：拒绝保存。
- 已停用偏好不会被后续自动学习重新开启。
- 已生效偏好才进入正文上下文；候选不影响故事。
- 撤回或重试回合时，后端按“存档名 + 回合 ID”移除被回溯回合贡献的证据并重新计算条目；不会用旧存档快照覆盖其他故事后来积累的全局偏好。

这些阈值是 MVP 的保守启发式，不代表数学上的真实概率。后续应根据真实日志调整。

## 数据范围

当前只有一张全局用户偏好卡，保存在 `data/preferences/user_preferences.yml`。每条证据仍记录来源故事，为后续区分以下作用域保留基础：

- 全局玩家偏好；
- 当前玩家角色偏好；
- 特定世界或故事偏好。

首版不自动把三者混合。角色偏好与故事隔离需要在后续版本单独设计。

## 后续工坊

偏好工坊用于对话式整理、合并、纠错和解释，不替代渐进学习。工坊应能展示证据来源，并让 AI 提议修改；所有大范围合并、反转和删除必须由玩家确认。

## 角色提取相关设计（后置）

“进入实体库”和“提取角色卡”是两个正交动作：

- 普通 NPC 可以被玩家关注并提取为本局/全局角色卡，而不参与 Overseer。
- 具备持续动机、幕后行动能力或长期影响的人物可以升格为实体，不论敌对与否。
- 同一人物可以通过稳定 ID 同时关联角色档案与实体状态。

玩家后期突然喜欢某个路人时，采用按需回溯：从永久保存的完整正文按名称、别名和相邻回合检索，再结合记忆摘要与可公开的实体历史，由 AI 生成带来源回合的可确认人物草稿。MVP 允许玩家补充别名和选择回合范围；语义检索后置。人格资料、故事记忆、本局状态和实体链接必须分离，跨世界载入时由玩家选择携带哪些记忆。

## 风险与待确认

- 相似但措辞不同的新观察，首版可能形成重复候选；语义合并不能在缺少确认时自动执行。
- 正文模型可能错误地把角色行为解释为玩家偏好，需要用真实日志持续校准提示词。
- 偏好中可能包含敏感内容；默认只保存在本地，未来任何同步或云端分析必须单独授权。
- 生图、世界书工坊、角色工坊和文风工坊如何各自消费偏好，需要逐个定义，不允许把整张偏好卡无差别塞给所有模型。
