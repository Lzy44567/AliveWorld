# 生图功能组设计

## 目标

AliveWorld 的生图功能首先服务于沉浸体验，而不是替代正文或要求玩家掌握 ComfyUI。首版同时支持角色立绘和剧情 CG，以本地 ComfyUI 为完整实现，并从底层预留云服务商适配接口。

生图必须与正文线程解耦：玩家触发任务后可以继续游戏，正文对应位置立即出现生成中占位卡；图片完成后原位替换、加入本局画廊，并弹出完成通知。

## 已确认决策

- 默认只允许玩家手动触发；“AI 自动判断是否生图”保留领域接口，但首版不实现判断逻辑，也不默认开启。
- 同时支持角色立绘、角色 CG 和场景 CG，但共用任务、服务商、存储和画廊基础设施。
- ComfyUI 使用常见默认地址 `http://127.0.0.1:8188`，必须允许修改、检测连接并报告实际错误。
- AliveWorld 内置至少一个仅依赖 ComfyUI 核心节点的 API 工作流，方便测试和新用户起步；同时允许导入、保存、配置用户工作流。
- 首版只完整接通 ComfyUI。云服务商先定义统一协议和 URL/鉴权配置，不假装已经支持未实现的厂商。
- R18 内容不得在本地失败后静默回退到云服务。任何云端上传都必须由玩家主动选择服务商，并遵守该服务商规则。
- AliveWorld 不自动安装 ComfyUI、Torch、模型或第三方节点，不在启动流程中隐式下载大型依赖。
- 提供不依赖 LLM 的接口测试入口，使用极简提示词生成测试图，并分别显示连接、提交、执行和取回结果是否成功。
- 支持角色参考图、画风参考图和玩家提供的风格偏好提示；参考图属于用户资产，必须明确关联用途，不直接塞进角色卡或正文文本。

## 两条工程线

### 代码工程

负责确定性流程：任务创建、持久化队列、状态流转、ComfyUI 提交与查询、取消/重试、图片落盘、消息关联、画廊、通知、日志和错误分类。

### 提示词工程

负责非确定性转换：从正文、角色卡、世界书、文风和玩家要求中整理画面，生成正向/负向提示词，并在未来接入用户偏好卡。提示词编译失败不能破坏任务系统，玩家始终可以绕过 LLM 直接填写生图提示词。

两条工程线通过结构化 `ImagePromptSpec` 交接，不让服务商适配器理解故事正文，也不让提示词模块操作 ComfyUI 队列。

`ImagePromptSpec` 可以包含画面尺度/表现偏好，但不设置一个与故事割裂的“本张图片是否 R18”开关。提示词编译器应从正文、角色和玩家要求理解内容，再根据玩家选择的表现尺度调整构图与描写重点。尺度枚举和措辞在提示词工程切片中另行验证，不能把模糊等级直接硬编码为所有模型通用规则。

## 核心流程

1. 玩家从角色卡、角色详情或某条正文消息发起生图。
2. 后端保存不可变的上下文快照：存档 ID、来源消息 ID、图片意图、相关角色、玩家要求和必要的世界书/文风摘要。
3. 前端立即在正文来源消息下插入任务占位卡；角色立绘任务也进入任务中心。
4. 玩家选择直接提示词，或让独立 LLM 编译为 `ImagePromptSpec`。
5. 持久化任务进入服务商队列；ComfyUI 适配器注入工作流参数并提交任务。
6. 后端查询或接收进度，前端更新占位卡，不阻塞正文生成和发送。
7. 成功结果复制到 AliveWorld 管理的本局图片目录，写入元数据并加入本局画廊。
8. 正文占位卡原位变为图片，弹出“图片已生成”通知；失败则保留错误、重试和编辑提示词入口。

## 任务状态

首版状态统一为：

`queued -> compiling_prompt -> ready -> submitted -> running -> succeeded`

任意可执行阶段都可以进入 `failed` 或 `cancelled`。应用重启后，未完成任务不得消失；无法确认远端状态的任务标为可恢复错误，不伪报成功。

任务至少记录：

- `id`、`save_id`、`intent`、`source_message_id`
- `provider_id`、`workflow_id`、工作流版本
- 上下文快照和最终 `ImagePromptSpec`
- 状态、进度、错误代码、创建/完成时间
- 输出图片路径、尺寸、seed、相关角色和参考图

## 图片意图

### 角色立绘

- 与角色卡或局内角色关联，可选设为当前立绘。
- 支持用户直接描述、AI 编译提示词和参考图。
- 差分属于后续切片；首版数据结构允许记录同一角色的多张图片，但不自动推断差分集合。

### 角色 CG

- 绑定一条正文消息及其中的一个或多个角色。
- 强调当前动作、服装、表情和场景，不自动覆盖角色立绘。

### 场景 CG

- 绑定正文消息，以场面和环境为主体；存在已知重要角色时允许携带角色参考图。

## 服务商适配

统一接口只暴露：连接检查、能力描述、提交、查询、取消和获取结果。具体工作流节点映射仅属于 ComfyUI 适配器。

### ComfyUI

- 默认地址为 `http://127.0.0.1:8188`，可配置。
- 内置工作流只能使用核心节点，并把 checkpoint、正负提示词、尺寸、步数、CFG、seed 等暴露为配置映射。
- 内置工作流不能假设用户一定拥有某个模型；连接后读取可用 checkpoint，要求玩家选择并保存。
- 用户导入的是 ComfyUI 的 API 格式工作流 JSON。导入时校验 JSON、节点和参数映射，原文件不直接执行任意本地脚本。
- 工作流配置按用户资产保存，局内设置只保存引用和覆盖参数。

### 云服务

首版仅实现适配协议和配置模型，不提供“通用 URL 就一定能生图”的虚假能力。每个真实云服务仍需独立适配器处理鉴权、提交格式、轮询和内容政策。

## 存储与画廊

- 图片是个人资产和游玩产物，保存在 `data` 范围内并遵守现有 Git 忽略规则。
- 每局拥有独立画廊；全局画廊只做跨存档汇总索引，不复制图片。
- 删除画廊图片前检查正文、角色立绘和任务引用；首版优先采用明确提示，不静默留下断链。
- 元数据与图片分离保存，避免只凭文件名恢复关系。

## 设置边界

全局设置提供默认服务商、ComfyUI 地址、默认工作流和默认生成参数；局内设置可以覆盖这些会改变本局体验的选项。

预留但首版不可用的“AI 自动判断是否生图”不能出现一个看似可用却没有行为的开关。自动判断需要单独设计触发条件、频率预算、重复画面抑制、R18 边界和玩家否决机制。

## v1.4 切片

1. 领域模型、持久化任务仓储、服务商协议和状态查询 API。
2. ComfyUI 连接检测、可用 checkpoint 查询、内置核心工作流和用户工作流导入。
3. 手动剧情 CG：正文占位卡、异步生成、原位替换、通知和失败重试。
4. 手动角色立绘：角色关联、参考图、设为当前立绘。
5. 本局画廊与跨存档汇总入口。
6. 独立提示词编译器和可编辑预览；始终保留直接提示词模式。

每个切片分别验证 Python 语法/测试、前端构建，以及使用模拟 ComfyUI 的行为测试。真实 ComfyUI 联调放在核心链路完成后进行人工验收。

## 2026-07-15 交互收敛决策

- 玩家端统一称为“生图模型”；`checkpoint` 仅保留为 ComfyUI 协议和文件类型术语。
- 默认入口改为“一键 AI 整理并生成”：点击后立即建立持久化任务，在后台调用提示词编译器，再自动提交 ComfyUI；正文线程不等待该流程。玩家填写的是自然语言补充要求，不要求掌握模型标签。
- 保留“直接使用当前提示词”作为次级入口，供熟悉模型标签的玩家绕过 LLM。
- 常用尺寸与数量直接可选，精确宽高、步数和 CFG 收入高级设置。高分辨率、多张和高步数必须提示显存与耗时成本。
- 模型适配采用“文件名 + 玩家可编辑的模型特性说明”，不凭 WAI、Illustrious、NoobAI 等文件名硬编码未经验证的能力。后续可提供可导入模型配置预设。
- 角色立绘提示词必须按可长期复用的角色视觉资产编译；角色 CG 和场景 CG 仍以当前正文为中心。
- 同一正文下的多图采用横向优先缩略网格，并允许点击放大；正文和画廊均提供重生成与安全删除。
- 全局角色立绘复制为全局角色资产，不能引用某个存档的任务路径；局内角色立绘继续使用本局任务关联。
- 参考图目前只完成安全存储与任务关联。只有工作流明确映射参考图节点后，才能宣称参考图参与生成。
