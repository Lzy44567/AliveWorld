# 01 活的世界：暗流实体 MVP

## 目的

使世界在玩家视野之外持续变化，但不强制每个回合制造事件，也不把所有人物转化为持久化 AI 人格。

## 数据契约

```yaml
name: 世界推演
motive: 根据世界书、世界日志和近期剧情推动世界自然演化
status: 低强度运行，观察当前故事规模
recent_actions:
  - 推演附近城镇因暴雨导致商路延误
plans:
  - 玩家前往受影响城镇时，商旅减少、物价上涨
mechanisms:
  - 小故事仅在明显相关时低频推演
  - 大世界可依据世界书与世界日志更积极推演
triggers:
  - condition: 玩家进入受影响城镇
    result: 商路延误的后果进入剧情
relationships:
  玩家: 间接影响
importance: 0.6
is_active: true
tags:
  - 世界推演
  - 暗流实体
```

字段含义：

- `motive` 是长期动机；不是一次行动的替代品。
- `status` 描述当前态势。
- `recent_actions` 只保留有限近期记录。
- `plans`、`mechanisms`、`triggers` 均是 AI 可理解的数据，不应被 Python 转写成世界观专属的硬编码 `if`。
- `relationships` 表达实体之间或与玩家的关系；MVP 可仅存储，不要求图谱 UI。
- `importance` 用于后续注意力控制；MVP 不据此实现复杂多阶段调度。

## 单回合流程

1. 玩家行动进入主线的未来发散与随机选择。
2. 主线结算生成正文和状态变化。
3. Overseer 读取世界书、近期剧情、影子账本、活跃实体摘要。
4. Overseer 可返回空动作；如有动作，更新实体、创建符合条件的新实体，或记录未来触发线索。
5. 更新写入本局实体库与影子账本。
6. 下一回合主线只注入受限长度的相关账本信息，使后果在合适场景出现。

## Overseer 的职责边界

Overseer 是统筹者，不是“默认天道”。它负责判断是否需要实体行动、哪些实体值得行动、输出怎样的状态与账本变化。

MVP 由一个 Overseer 一次性统筹少量实体。实体达到 12 个活跃项时再评估筛选；超过 20 个活跃项时才将“先筛选、后逐个推演”列为必要改造。

## 注意力与剧透控制

注入按相关性分层：

- 活跃实体：完整或接近完整的信息。
- 相关实体：短摘要。
- 非相关实体：不注入，只留在本局实体库。

可见性模式：`hidden`、`names`、`motives`、`full`。`forecast` 只表示倾向、预兆或可能性，不能冻结“下一回合剧本”，列为 P3。

## 约束

- 世界书规则优先于暗流实体行动，暗流实体行动优先于主线即兴发挥。
- 角色卡与普通 NPC 不自动进入实体库。
- 世界推演通过实体的 `is_active` 参与或退出，不新增独立全局开关表。
- 小故事的低频与低强度应由 Overseer 的判断和提示词约束，不先写死“每 N 回合一次”。

## 暂缓问题

1. 大世界/小故事由玩家选择、世界书复杂度估算，还是两者结合？
2. 玩家是否可手动将重要 NPC 升格为暗流实体？
3. 实体是否可彼此行动，以及该结果如何写入账本？
4. 最小 UI 采用已有调试入口扩展，还是先只提供后端能力与调试文本？
