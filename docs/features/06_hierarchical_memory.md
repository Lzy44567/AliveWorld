# 分级记忆与上下文管理（设计草案）

> 状态：首版 Alpha 已实现，等待真实长故事人工验证。
>
> 来源优先级：原始手稿 9.10、用户 2026-07-16 补充设计。

## 目标

让长篇故事在不无限增加模型输入的前提下保持事件连贯、重要事实与未解决线索。压缩只改变发给 AI 的工作上下文，不删除完整原文。

## 已确认边界

- 完整玩家输入、正文和结构化回合结果必须永久保留，可审计、可恢复。
- 世界书、角色卡、实体库和暗流因果账本不进入故事摘要压缩；它们继续按各自规则触发和注入。
- token 预算只触发“压缩检查”，不允许在连贯事件中间机械切断。
- 压缩模型可独立配置；留空时使用正文 LLM 配置。本地模型通过兼容接口接入。
- 手动压缩不作为默认游玩步骤；首版若保留，放入调试/高级工具。

## 分级记忆草案

1. **即时记忆**：最近的完整回合原文，总是直接注入。
2. **日常记忆**：已结束事件或安全切片的详细摘要，按顺序单独存档，不按现实日期强行分割。
3. **重点记忆**：转折、承诺、持久变化、未解决线索和长期因果；比日常记忆更高优先级注入。
4. **核心记忆**：影响整个故事身份与长期连续性的极少数事实，默认常驻，不因普通再压缩丢失。

这些名称是记忆优先级，不是四份重复的完整摘要。世界规则、角色卡状态、实体行动和因果账单仍归各自模块管理，避免内容所有权冲突。

## 触发与安全切片

1. Python 使用可替换的 token 估算器统计当前工作上下文。阈值不使用一个适用所有模型的固定数字，而是从模型上下文上限扣除输出预留、系统提示词、当前资产和安全余量后得出剧情历史预算。
2. 使用高/低水位：工作历史达到高水位才触发检查，成功切片后降到低水位，避免每新增少量文字就再次压缩。
3. 达到高水位后只标记“需要压缩”；若采用事件状态方案，则优先寻找已经结束的事件边界，否则按固定 token 与热窗口规则选择保守切片位置。
4. 始终保留一段未压缩的热窗口；即使触发阈值，也不把刚开始的短事件切入摘要。
5. 没有已结束事件可供切分时，不无限等待；达到硬上限后使用“当前连续性胶囊”保留地点、参与者、目标、当前冲突与未完成动作，再按固定 token 保守切片。

### 候选方案 A：正文 AI 维护事件状态

用户期望正文 AI 在写完正文后，同次返回事件名称、事件核心目标和状态，而不是只返回模糊的软/硬边界。候选结构：

```json
"event_updates": [
  {
    "event_id": "event_由后端分配或复用",
    "name": "寻找失踪的商队",
    "core_goal": "确认商队下落并处理造成失踪的直接原因",
    "status": "active | completed | paused",
    "completion_evidence": ""
  }
]
```

- 新事件创建时必须给出相对稳定、可验证的 `core_goal`，不能只写“推进剧情”“继续生活”等空目标。
- 只有正文中已经发生了满足核心目标的事实，才允许标记 `completed`；同时给出简短的 `completion_evidence` 供后端和压缩任务校验。
- `paused` 表示事件暂时离开当前叙事焦点但尚未完成，不能被当作安全结束边界。
- 没有明确事件的日常叙事允许返回空数组，禁止为了填充字段而强行创造任务、冲突或转折。
- 主线、支线、平行事件和子事件如何建模尚未确认；首版不能在未设计清楚时直接扩张成复杂事件图。

该方案的主要成本不是额外 API 请求，而是每回合要把精简的活动事件索引传给正文 AI，并由正文 AI 返回事件变更。主要风险是误判完成、事件漂移和把日常故事任务化。

### 推荐首版：token 驱动的前缀压缩

首版不必依赖正文 AI 每回合维护事件。Python 在未压缩原文达到高水位后，把较早的完整回合前缀压缩到目标低水位，并始终保留足够大的近期热窗口。token 只决定何时压缩和大致压缩多少，不允许从一句话、一个消息或一个回合中间截断。

例如工作历史达到 10000 token 时，目标不是机械切在第 5000 个 token，而是从旧到新选择最接近目标的完整回合前缀；压缩后让未压缩热窗口回落到约 5000 token。具体数字只是示例，实际按模型上下文预算动态计算。

压缩任务读取：

- 本次准备归档的完整回合前缀；
- 与热窗口相接的一小段重叠桥接上下文；
- 上一版精简的未解决线程索引。

压缩输出至少区分：

- 已发生事实与因果摘要；
- 尚未完成的长期目标、承诺、冲突、悬念和已知进展；
- 已完成或失效的旧线程及其依据；
- 需要进入重点/核心记忆的候选事实。

这样，长期任务即使跨越多个压缩周期，也会作为“未解决线程”留在精简索引中；后续压缩任务再结合新片段更新或结束它。事件目标判断发生在低频压缩任务中，而不是每回合额外运行。

该方案会牺牲一部分天然事件边界，但实现简单、可预测，也不会诱导正文 AI 为元数据塑造剧情。正文事件状态系统可作为后续独立升级，并在成熟后帮助压缩器选择更自然的切片位置。

### 长期任务的内容归属

- 普通故事任务、玩家承诺和未解决剧情进入故事记忆的“未解决线程索引”，不写入世界书。
- 世界书只保存世界规则、背景、公理和可复用设定；只有长期任务造成了稳定世界设定变化，才由世界书捕获流程另行处理其结果。
- 实体库只保存实体自身的动机、计划、状态、行动与关联影响；某实体参与长期任务时，可以记录它自己的计划或行动，但不把整条玩家任务复制为实体库的第二份真相源。
- 暗流因果账本只保存会在幕后持续、等待条件或产生后果的因果影响，例如通缉、陷阱、传播中的瘟疫和秘密追杀机制；它不保存“玩家正在完成什么任务”或完整剧情进度。
- 当长期任务由某条暗流影响引起时，未解决线程只引用其 `influence_id`；影响是否有效、触发、取消或消耗仍以暗流因果账本为唯一真相源，避免复制后状态分叉。
- 后续若开发玩家可见的任务日志，应作为独立投影视图或引用故事线程 ID，避免世界书、实体库和记忆系统之间重复保存后逐渐冲突。

### 候选架构：复用账本生命周期引擎

暗流账本已有来源、创建回合、存在时长、条件、状态和触发次数等生命周期能力，适合抽成通用账本内核，并让故事事件复用。但不把暗流影响与故事事件混成同一种记录：

- `hidden_influence`：条件满足后产生或兑现影响，可一次性消耗或持续存在，默认由 Overseer 与暗流系统管理。
- `story_event`：拥有事件名称、核心目标、完成条件、进展与状态；条件满足后把事件标记为完成，不凭空产生一次影响。

二者共享 ID、来源引用、创建回合、累计回合、状态流转、撤回回滚和审计日志；分别保留不同的字段、提示词、可见性、AI 责任和结算动作。不要把 `trigger_condition` 直接改成 `event_goal`，而是在共享内核上分别使用 `trigger_condition` 与 `completion_condition`。

若首版只为压缩服务，`story_event` 可由低频压缩任务创建和更新，允许完成状态延迟到压缩检查时确认；若以后要实时任务 UI 或立即沿事件边界切片，再让正文 AI 同次返回事件增量，不额外创建每回合 LLM 请求。

### 调用频率：不新增每回合记忆 LLM

- 即使采用事件状态方案，每回合也仍只使用现有正文 LLM；事件更新作为正文 JSON 字段同次返回，不新增请求。
- 压缩 LLM 只在 token 高水位触发时运行；不每回合访问。
- 压缩请求只读取“本次待压缩的原文片段 + 已有精简记忆/未解决线索索引”，不重新上传整部故事。
- 故事线程的归并、新增、暂停与解决发生在这次压缩任务中；正文 AI 不负责每回合维护完整线程库。

## 摘要持续膨胀的处理

不把所有历史摘要永久全量拼接。“新原文切片”与“旧摘要整理”使用两个不同触发器：

- 只有上次成功切片后新增的未压缩原文达到预算，才创建新事件切片；旧摘要自身不反复触发同一任务。
- 只有当前实际注入的摘要达到记忆子预算高水位，才启动章节脊柱整理或注入筛选。

首选固定层次而不是无限递归压缩：

- 保留所有日常/事件摘要文件作为可追溯记录。
- 较旧事件摘要合并为有长度上限的章节脊柱，而不一直追加。
- 核心记忆常驻；重点记忆按未解决状态和当前相关性选择注入。
- 较旧的详细摘要不默认全量注入，后续可复用关键词/语义检索按需回忆。

## 当前实现逻辑（v1.4.4 Alpha）

### 正文上下文接管

- `GameSession` 为每个成功回合写入单调增长的 `turn_id`、玩家行动、正文、token 估算和对应聊天消息 UUID；完整聊天记录与旧 `context_history` 继续保留。
- 开启自动压缩时，正文、世界书检索和暗流所需的近期故事文本统一由 `StoryMemoryManager.build_context()` 组装，不再只读取最后 3 段。
- 关闭自动压缩时仍保留旧版“最后 3 段”行为，避免玩家关闭压缩后无上限发送历史或突然改变旧存档成本。
- 压缩尚未完成但原文已经超过预算时，临时优先发送最新、未超过历史高水位的完整回合；成功压缩后自动切换为“长期记忆 + 未解决事件 + 近期完整回合”。

### token 预算与切片

- 当前使用不增加依赖的中英混合保守估算器：中文字符近似逐字计数，拉丁数字按约 4 字符一个 token，标点计入安全余量。它用于水位控制，不冒充供应商账单中的精确 token。
- 玩家填写压缩模型上下文上限；当前历史高水位取上限的 38%（限制在 3000～32000），目标低水位取高水位的 55%，桥接上下文取高水位的 10%（限制在 500～2400）。默认上下文上限为 32768。
- 长期记忆实际注入另设约为上下文上限 18% 的子预算（限制在 1500～8000 token 估算）；核心记忆、活动事件、章节脊柱、重点记忆和近期摘要按优先级装入，不让摘要增长反过来挤掉全部近期原文。
- 切片只选择完整 `story_turn`；始终保留至少 4 个近期完整回合。手动调试压缩在回合较少时至少保留约一半，不从消息中间截断。
- 具体比例集中在 `MemoryBudget`，不是散落在会话代码中的常量；真实测试后可替换而不改变存储协议。

### 低频压缩与事件更新

- 每回合开始和成功保存后只做 Python 水位检查；未达到水位时不会调用压缩模型。
- 达到水位后启动 daemon 后台线程。压缩请求只读取本次旧回合前缀、少量桥接上下文、上一版章节脊柱和故事事件索引。
- 同一次压缩输出片段摘要、更新后的有界章节脊柱、重点/核心记忆候选，以及发生变化的 `story_event`。
- `story_event` 目前由低频压缩模型维护，字段包括名称、核心目标、完成条件、进展、状态、来源回合和关联暗流影响 ID；没有完成依据时，Python 拒绝把事件改成 `completed`。
- 当前没有把现有暗流账本重构成通用父类。故事事件采用独立领域文件并保持兼容字段；待真实需求稳定后再抽共享生命周期内核，避免为抽象而破坏已验证的暗流闭环。

### 失败、撤回与原子性

- 后台压缩失败只写本地日志和状态，不推进 `archived_until_turn`，不覆盖上一份有效索引，也不影响本回合正文。
- 片段与索引使用临时文件后原子替换；先保存含完整原文的片段，再发布新索引。
- 普通撤回恢复 `story_turns`，并继续复用暗流账本现有快照。后台压缩只归档热窗口之前的旧回合，因此不会把可撤回的最近回合纳入片段。
- `memory/index.json` 是运行时记忆真源，`session_state.json` 同时保存索引快照用于兼容回退；存档克隆会随目录复制完整记忆文件。

### 模型与调试入口

- 全局设置新增压缩模型 Base URL、API Key、模型名和上下文上限；三个模型字段全部留空时复用正文 `AIEngine`。填写独立地址但不填密钥时使用无敏感信息的本地占位密钥，不把正文 API Key 转发给本地服务。
- 局内设置保留独立自动压缩开关。调试区可以刷新水位、片段数、活动事件数和最近错误，并手动执行一次完整回合压缩；手动入口不属于默认游玩流程。
- 完整压缩请求和响应进入现有本地 LLM Trace，方便核对真正发送了哪些原文、事件和摘要。

## 其他可替换实现逻辑

- **精确 tokenizer**：以后可按模型选择 tokenizer 或读取供应商 usage；当前估算器保持零新增大型依赖和跨供应商可运行。
- **正文实时事件增量**：如果低频更新不能满足任务 UI 或自然边界，可让正文同次 JSON 返回事件增量；仍不新增每回合事件 LLM。
- **通用账本内核**：暗流影响和故事事件可以以后共享 ID、时长、回滚与审计基础设施，但必须保留不同提示词、权限和结算动作。
- **按需回忆**：旧片段完整保存在独立文件；首版不做关键词/向量召回，后续可以只改变检索器，不改变原始片段格式。
- **精细摘要再压缩**：当前每次压缩都重写有 8000 字符上限的章节脊柱，并只注入最近 2 个详细片段摘要；以后可拆成独立的章节整理触发器。

## 存储协议

- 完整原文保留在存档的追加式记录中；不以现实“每天”作为故事强制分割单位。
- 每个完整回合拥有存档内单调增长的数字 `turn_id`；现有消息 UUID 继续用于精确消息引用。
- `memory/segments/turn_000021-000037.json`：文件名直接表示源回合 ID 范围，内部保存每个完整回合、对应消息 UUID、token 估算和片段摘要。
- `memory/index.json`：保存当前热窗口边界、章节脊柱、重点/核心记忆索引、版本和来源。
- 文件原子替换；压缩失败时继续使用上一份有效上下文，不覆盖原文或有效索引。

## 独立模型与运行方式

- 压缩设置支持独立 Base URL、API Key 和模型名；全部留空则继承正文模型。
- 摘要不需要剧情创作能力，但需要足够的长上下文、事实保真和结构化输出能力；不能默认“越小越好”。
- 压缩任务在回合保存后运行，成功后原子切换新记忆版本；任务未完成时下一回合继续使用旧的有效版本。

## 待确认

1. 高/低水位、热窗口和桥接区的当前默认比例是否适合真实中文长故事，需要通过不同模型和资产规模验证。
2. 重点/核心记忆是否允许玩家查看和编辑；首版仅在索引、上下文和日志中可审计。
3. 旧记忆的按需回忆何时进入版本计划；首版只注入章节脊柱、活动事件、重点/核心记忆和最近片段。
4. 是否需要实时任务/事件 UI；首版事件状态允许延迟到下一次压缩时更新。
