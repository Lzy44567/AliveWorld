# 分级记忆与上下文管理（设计草案）

> 状态：讨论中，尚未进入代码开发。
>
> 来源优先级：原始手稿 9.10、用户 2026-07-16 补充设计。

## 目标

让长篇故事在不无限增加模型输入的前提下保持事件连贯、重要事实与未解决线索。压缩只改变发给 AI 的工作上下文，不删除完整原文。

## 已确认边界

- 完整玩家输入、正文和结构化回合结果必须永久保留，可审计、可恢复。
- 世界书、角色卡、实体库和暗流因果账本不进入故事摘要压缩；它们继续按各自规则触发和注入。
- token 预算只触发“压缩检查”，不允许在连贯事件中间机械切断。
- 压缩模型可独立配置；留空时使用正文 LLM 配置。本地模型通过兼容接口接入。
- 手动压缩不作为默认游玩步骤；首版若保留，放入调试/高级工具。

## 分级记忆草案

1. **即时记忆**：最近的完整回合原文，总是直接注入。
2. **日常记忆**：已结束事件或安全切片的详细摘要，按顺序单独存档，不按现实日期强行分割。
3. **重点记忆**：转折、承诺、持久变化、未解决线索和长期因果；比日常记忆更高优先级注入。
4. **核心记忆**：影响整个故事身份与长期连续性的极少数事实，默认常驻，不因普通再压缩丢失。

这些名称是记忆优先级，不是四份重复的完整摘要。世界规则、角色卡状态、实体行动和因果账单仍归各自模块管理，避免内容所有权冲突。

## 触发与安全切片

1. Python 使用可替换的 token 估算器统计当前工作上下文。阈值不使用一个适用所有模型的固定数字，而是从模型上下文上限扣除输出预留、系统提示词、当前资产和安全余量后得出剧情历史预算。
2. 使用高/低水位：工作历史达到高水位才触发检查，成功切片后降到低水位，避免每新增少量文字就再次压缩。
3. 达到高水位后只标记“需要压缩”；若采用事件状态方案，则优先寻找已经结束的事件边界，否则按固定 token 与热窗口规则选择保守切片位置。
4. 始终保留一段未压缩的热窗口；即使触发阈值，也不把刚开始的短事件切入摘要。
5. 没有已结束事件可供切分时，不无限等待；达到硬上限后使用“当前连续性胶囊”保留地点、参与者、目标、当前冲突与未完成动作，再按固定 token 保守切片。

### 候选方案 A：正文 AI 维护事件状态

用户期望正文 AI 在写完正文后，同次返回事件名称、事件核心目标和状态，而不是只返回模糊的软/硬边界。候选结构：

```json
"event_updates": [
  {
    "event_id": "event_由后端分配或复用",
    "name": "寻找失踪的商队",
    "core_goal": "确认商队下落并处理造成失踪的直接原因",
    "status": "active | completed | paused",
    "completion_evidence": ""
  }
]
```

- 新事件创建时必须给出相对稳定、可验证的 `core_goal`，不能只写“推进剧情”“继续生活”等空目标。
- 只有正文中已经发生了满足核心目标的事实，才允许标记 `completed`；同时给出简短的 `completion_evidence` 供后端和压缩任务校验。
- `paused` 表示事件暂时离开当前叙事焦点但尚未完成，不能被当作安全结束边界。
- 没有明确事件的日常叙事允许返回空数组，禁止为了填充字段而强行创造任务、冲突或转折。
- 主线、支线、平行事件和子事件如何建模尚未确认；首版不能在未设计清楚时直接扩张成复杂事件图。

该方案的主要成本不是额外 API 请求，而是每回合要把精简的活动事件索引传给正文 AI，并由正文 AI 返回事件变更。主要风险是误判完成、事件漂移和把日常故事任务化。

### 推荐首版：token 驱动的前缀压缩

首版不必依赖正文 AI 每回合维护事件。Python 在未压缩原文达到高水位后，把较早的完整回合前缀压缩到目标低水位，并始终保留足够大的近期热窗口。token 只决定何时压缩和大致压缩多少，不允许从一句话、一个消息或一个回合中间截断。

例如工作历史达到 10000 token 时，目标不是机械切在第 5000 个 token，而是从旧到新选择最接近目标的完整回合前缀；压缩后让未压缩热窗口回落到约 5000 token。具体数字只是示例，实际按模型上下文预算动态计算。

压缩任务读取：

- 本次准备归档的完整回合前缀；
- 与热窗口相接的一小段重叠桥接上下文；
- 上一版精简的未解决线程索引。

压缩输出至少区分：

- 已发生事实与因果摘要；
- 尚未完成的长期目标、承诺、冲突、悬念和已知进展；
- 已完成或失效的旧线程及其依据；
- 需要进入重点/核心记忆的候选事实。

这样，长期任务即使跨越多个压缩周期，也会作为“未解决线程”留在精简索引中；后续压缩任务再结合新片段更新或结束它。事件目标判断发生在低频压缩任务中，而不是每回合额外运行。

该方案会牺牲一部分天然事件边界，但实现简单、可预测，也不会诱导正文 AI 为元数据塑造剧情。正文事件状态系统可作为后续独立升级，并在成熟后帮助压缩器选择更自然的切片位置。

### 长期任务的内容归属

- 普通故事任务、玩家承诺和未解决剧情进入故事记忆的“未解决线程索引”，不写入世界书。
- 世界书只保存世界规则、背景、公理和可复用设定；只有长期任务造成了稳定世界设定变化，才由世界书捕获流程另行处理其结果。
- 实体库只保存实体自身的动机、计划、状态、行动与关联影响；某实体参与长期任务时，可以记录它自己的计划或行动，但不把整条玩家任务复制为实体库的第二份真相源。
- 后续若开发玩家可见的任务日志，应作为独立投影视图或引用故事线程 ID，避免世界书、实体库和记忆系统之间重复保存后逐渐冲突。

### 调用频率：不新增每回合记忆 LLM

- 即使采用事件状态方案，每回合也仍只使用现有正文 LLM；事件更新作为正文 JSON 字段同次返回，不新增请求。
- 压缩 LLM 只在 token 高水位触发时运行；不每回合访问。
- 压缩请求只读取“本次待压缩的原文片段 + 已有精简记忆/未解决线索索引”，不重新上传整部故事。
- 故事线程的归并、新增、暂停与解决发生在这次压缩任务中；正文 AI 不负责每回合维护完整线程库。

## 摘要持续膨胀的处理

不把所有历史摘要永久全量拼接。“新原文切片”与“旧摘要整理”使用两个不同触发器：

- 只有上次成功切片后新增的未压缩原文达到预算，才创建新事件切片；旧摘要自身不反复触发同一任务。
- 只有当前实际注入的摘要达到记忆子预算高水位，才启动章节脊柱整理或注入筛选。

首选固定层次而不是无限递归压缩：

- 保留所有日常/事件摘要文件作为可追溯记录。
- 较旧事件摘要合并为有长度上限的章节脊柱，而不一直追加。
- 核心记忆常驻；重点记忆按未解决状态和当前相关性选择注入。
- 较旧的详细摘要不默认全量注入，后续可复用关键词/语义检索按需回忆。

## 存储草案

- 完整原文保留在存档的追加式记录中；不以现实“每天”作为故事强制分割单位。
- 每个完整回合拥有存档内单调增长的数字 `turn_id`；现有消息 UUID 继续用于精确消息引用。
- `memory/segments/turn_000021-000037.json`：文件名直接表示源回合 ID 范围，内部另存首尾消息 UUID、token 数、详细摘要和分级记忆候选。
- `memory/index.json`：保存当前热窗口边界、章节脊柱、重点/核心记忆索引、版本和来源。
- 文件原子替换；压缩失败时继续使用上一份有效上下文，不覆盖原文或有效索引。

## 独立模型与运行方式

- 压缩设置支持独立 Base URL、API Key 和模型名；全部留空则继承正文模型。
- 摘要不需要剧情创作能力，但需要足够的长上下文、事实保真和结构化输出能力；不能默认“越小越好”。
- 压缩任务在回合保存后运行，成功后原子切换新记忆版本；任务未完成时下一回合继续使用旧的有效版本。

## 待确认

1. 是否正式确认首版采用 token 驱动的前缀压缩，并把正文事件状态系统后置为独立升级。
2. 高/低水位、热窗口和桥接重叠区的默认比例需通过不同模型上下文、中英文比例和资产规模测试后确定，高级设置是否允许玩家覆盖。
3. 重点/核心记忆首版是否允许玩家查看和编辑，还是先只在日志中调试可见。
4. 旧记忆的按需回忆是否进入首版，或先仅注入章节脊柱、当前未解决重点与最近事件。
