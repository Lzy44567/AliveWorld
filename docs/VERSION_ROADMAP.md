# v1.1.0 路线图：活的世界 MVP

## 版本与提交约定

- 作者/环境标签：`codex-gpt5.6sol`
- 下一个目标版本：`v1.1.0`
- 提交说明格式：`vX.Y.Z: <变更摘要> (codex-gpt5.6sol)`
- 发布时创建独立版本 Git tag：`v1.1.0`。

作者标签和版本 Git tag 承担不同职责：前者记录实现环境和模型，后者标识可回退、可复现的版本节点。除非用户明确要求，不在开发中途创建 tag 或推送远端。

## v1.1.0 目标

让少量暗流实体在每回合由一个 Overseer 统筹，形成可持久化、可触发、受世界书约束的世界变化，同时避免把普通 NPC 和角色卡卷入长期人格推演。

## 玩家可感知的完整闭环

v1.0.0 已经提供随机未来、世界书、角色/文风/实体资产、状态栏与极简暗流。v1.1.0 要补齐它们之间的因果链，而不是重新发明一套平行系统：

1. 玩家载入世界书、角色卡、文风和必要的暗流实体，开始故事。
2. 世界书提供规则与触发条目；角色卡和普通 NPC 只服务当前剧情，不自动获得长期幕后人格。
3. 每回合主线仍负责“多个未来 → 权重 → 随机选中 → 正文和状态结算”。
4. 正文完成后，Overseer 结合世界书、近期剧情、活跃实体和影子账本，判断本回合是否有幕后变化；它可明确返回“无动作”。
5. 若有变化，实体更新状态、计划、机制或触发线索，写入本局实体库与影子账本。
6. 后续主线只在相关场景读取受限的暗流摘要，玩家通过事件后果、传闻、物价、阻拦或人物行动感知世界变化，而非被剧透文本打断。
7. 调试/玩法可见性决定玩家能看到实体名称、动机或完整信息；默认不泄露暗流。

“世界推演”是可启用或封存的系统型实体，用于让大世界持续演化；它不是默认天道、最终反派或独立世界状态表。

## v1.1.0 非目标

- 独立 `world_state` 表
- 全量 NPC 社会模拟或人格持久化
- 世界书自动演化
- `forecast` 预知玩法
- 关系图 UI、画图、对话/群聊、用户偏好分析
- 大规模实体的异步或多阶段推演

## 当前状态

切片 1、2、3、4 与测试基础设施均已完成并提交，v1.1.0 活的世界 MVP 的计划范围已闭环。下一阶段应以实际游玩反馈为依据，优先修复因果触发、上下文长度和 UI 可理解性问题；不将旧 redesign 代码或 patch 直接合并到当前基准。

## v1.1 Alpha 的人工验收前收尾

### 目标

把 v1.1 已具备的暗流实体、资产开关和模板机制变成可被创作者验证、理解和维护的闭环，不扩大为 NPC 社会模拟或独立世界状态系统。

### 已进入实现的内容

1. `data/` 内模板标识与模板标签展示。
2. 实体编辑去 JSON 化。
3. 世界书、文风、角色卡、实体的关闭回归测试。

### 候选小功能

1. 调试态的“本回合上下文来源”摘要：只在授权显示时列出实际启用的世界书条目、文风、角色卡和实体，便于定位为什么某张卡生效。
2. 实体局内卡的只读状态摘要：状态、最近行动和待触发线索，受现有名称/动机/编辑权限约束。
3. 模板卡与个人卡的更明确区分、模板复制/覆盖提示。

### 人工验收门槛

在上述三个候选小功能完成后进行第一次完整人工验收。不要再等到世界书自动演化、forecast 或 NPC 社会模拟；那些属于后续版本，继续堆叠只会扩大定位成本。

## 版本策略

- Git 提交：每个可验证切片一条提交，说明中保留作者/环境标签。
- Git tag：只在人工验收通过的发布节点创建，不为每次开发或每个小功能创建 tag。
- `vX.Y.Z`：补丁修复用 `Z`，一个完整且可感知的功能阶段用 `Y`；只有整体产品体验、架构或核心循环发生跃迁时使用新的 `X`。

当前工作属于 `v1.1` 活世界 Alpha；在未完成完整人工验收前不声明新的正式版本。

## v1.4.0：世界推演职责收口与生图基础版

世界书工具链、动态未来和暗流因果账本已具备可运行 MVP。`v1.4.0-dev.1` 已让“世界推演”承担宏观世界演化职责，并明确普通实体的能力和信息边界。

第一切片只建立职责协议：实体摘要向 Overseer 提供类型、标签和定位；系统型世界推演关注制度、经济、势力、地区环境、灾难与社会风貌，普通实体仍受自身能力与信息限制。世界推演允许连续多回合无动作，不围绕玩家制造巧合、特殊道具或陷阱，也不能改写世界书公理。

世界推演在此收口：固定每 N 回合运行、自动判断大世界/小故事、强度滑块、独立世界状态表、概率衰减或多阶段实体调度，继续等待后续设计和真实日志证据。

重新规划后的 v1.4 主开发线转为生图基础版：完整接通本地 ComfyUI，同时提供可扩展云服务协议；实现手动角色立绘、剧情 CG、持久化异步任务、正文占位/原位替换、通知和本局画廊。AliveWorld 内置可配置的核心节点工作流，并允许导入用户工作流。AI 自动判断生图、自动用户偏好学习、动图和自动安装模型不进入本版本。详见 [`features/05_image_generation.md`](features/05_image_generation.md)。

## v2.0 方向：整体体验与核心亮点升级

v2 不是“实体推演单功能升级”。它需要同时让玩家感受到世界、资产和沉浸体验的系统级提升。候选支柱如下，具体开工前逐项做定向需求访谈：

1. 活的世界：暗流实体与世界推演从 Alpha 走向稳定、可观测、可触发的长期因果。
2. 世界书演化：在明确开关和边界下，识别并扩展故事中新出现的世界设定；不能替代世界书规则，也不能无控制膨胀。
3. 生图功能组：角色立绘、CG、画廊与异步任务基础；这是高成本系统，需要单独设计工作流、失败恢复和资产管理。
4. 用户偏好卡：仅在用户明确授权、数据边界和更新规则明确后考虑；不作为当前 P0。

记忆管理是通用能力而不是当前核心卖点，暂列 P2；角色卡“玩家扮演/角色陪玩”的职责重构列为 P1，等开发该模块前再进行专门需求访谈。

## v1.5：渐进式用户偏好卡

- v1.5-dev.1：正文结算同次返回偏好观察；后端以证据次数、明确表达和可信度区分候选与已生效偏好，不新增独立每回合模型请求。
- v1.5-dev.2：正文改为只记录中性行为证据；低频分析模型提出竞争假设与序数证据强度，Python 用固定似然比计算后验；加入敏感分析开关、分类消费开关、独立模型配置、偏好元信息输入、待确认红点和版本标识。
- v1.5-dev.3：成功的撤回、重试、重掷与生图操作由 Python 记录为保守弱证据；偏好页新增证据时间线和后验依据，形成可快速检查的学习闭环。
- v1.5-dev.4：实现持久化用户偏好工坊，支持连续讨论、三种协作方向、提案与草稿权限、逐项高风险确认、手工编辑、撤销、发布及正式偏好卡并发更新后的安全重放。
- v1.5-dev.5：实现总工坊统一入口；世界书、角色、文风、实体和偏好均形成真实草稿闭环，角色/文风/实体使用共享事务内核与分类型 Agent 协议；记忆模型配置并入 API 配置。
- 提供全局偏好管理页，重点是检查与纠错，不把手工填表作为主要形成方式。
- 学习与正文注入分别提供全局默认和局内开关；关闭后对应提示词/上下文不再发送。
- 后续切片再设计跨故事作用域、语义去重、交互原因反馈，以及生图/各类工坊的按需消费协议。
- 角色工坊阶段接入普通 NPC 回溯提取与非敌对重要角色升格；不把该功能混入偏好卡首版。

## v1.4.2：行动建议与基础游玩闭环

- 正文结算同一次 JSON 返回 2-4 个玩家下一步行动建议，不新增固定 LLM 请求。
- 建议只能依据本轮正文、玩家可见状态和已知信息，不得泄露暗流账本、后台实体、未触发陷阱或未被选中的未来。
- 隐藏因果只供前置条件判断；剧情结算改用玩家可见世界上下文，并通过单独的“本回合必须兑现影响”接收已触发后果。
- 建议以 A/B/C/D 两列网格显示，点击直接发送；玩家也可输入 `A` 或 `选 B，再补充要求`，由后端展开为完整建议与补充内容。
- 关闭建议时完全不追加建议提示词或输出字段，避免无效上下文消耗。
- 建议随存档、撤回、重试和重掷恢复；局内开关关闭时后端拒绝保存模型误返回的建议。

聊天模式不是 v1.4.2 的界面换皮任务。它未来可以复用聊天气泡、输入与生图组件，但需独立设计角色扮演提示词、短消息返回协议、社交平台节奏、关系记忆、自拍/群聊与不推进正文世界的状态生命周期。

## v1.4.3：资产分支与版本实验

- 存档、角色卡、世界书、文风卡和实体卡提供一致的重命名/克隆入口。
- 克隆用于保留当前版本或尝试不同创作分支；副本内容独立，不静默覆盖同名目标。
- 模板不可重命名，但可克隆成个人资产。关联立绘先共享已有图片引用，后续重新生成/指定时只改动目标卡。
- 重命名会维护当前会话、实体因果引用和未发布工坊草稿；运行中生图任务会阻止存档目录迁移。

## v1.3 开始：动态未来候选与世界书边界

第一切片先优化现有单层随机未来：事实资格、关键依据、相对权重、Python 抽取和完整日志。它不实现多层 `N×n` 展开。

该切片稳定后，再进入世界书专项设计与开发：优先核验相关条目命中和冲突识别，然后设计“既有事实 / AI 演化草稿 / 玩家确认内容”的边界，不允许 AI 无审查覆盖既有规则。
