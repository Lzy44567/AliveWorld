# v1.1.0 路线图：活的世界 MVP

## 版本与提交约定

- 作者/环境标签：`codex-gpt5.6Terra`
- 下一个目标版本：`v1.1.0`
- 提交说明格式：`v1.1.0: <变更摘要> (codex-gpt5.6Terra)`
- 发布时创建独立版本 Git tag：`v1.1.0`。

作者标签和版本 Git tag 承担不同职责：前者记录实现环境和模型，后者标识可回退、可复现的版本节点。除非用户明确要求，不在开发中途创建 tag 或推送远端。

## v1.1.0 目标

让少量暗流实体在每回合由一个 Overseer 统筹，形成可持久化、可触发、受世界书约束的世界变化，同时避免把普通 NPC 和角色卡卷入长期人格推演。

## 玩家可感知的完整闭环

v1.0.0 已经提供随机未来、世界书、角色/文风/实体资产、状态栏与极简暗流。v1.1.0 要补齐它们之间的因果链，而不是重新发明一套平行系统：

1. 玩家载入世界书、角色卡、文风和必要的暗流实体，开始故事。
2. 世界书提供规则与触发条目；角色卡和普通 NPC 只服务当前剧情，不自动获得长期幕后人格。
3. 每回合主线仍负责“多个未来 → 权重 → 随机选中 → 正文和状态结算”。
4. 正文完成后，Overseer 结合世界书、近期剧情、活跃实体和影子账本，判断本回合是否有幕后变化；它可明确返回“无动作”。
5. 若有变化，实体更新状态、计划、机制或触发线索，写入本局实体库与影子账本。
6. 后续主线只在相关场景读取受限的暗流摘要，玩家通过事件后果、传闻、物价、阻拦或人物行动感知世界变化，而非被剧透文本打断。
7. 调试/玩法可见性决定玩家能看到实体名称、动机或完整信息；默认不泄露暗流。

“世界推演”是可启用或封存的系统型实体，用于让大世界持续演化；它不是默认天道、最终反派或独立世界状态表。

## 迭代任务

| 切片 | 目标 | 主要边界 | 验收 |
| --- | --- | --- | --- |
| 1. 实体领域边界 | 结构化实体、YAML 仓储、结构化账本、旧数据兼容 | `core/entities.py`、`core/entity_repository.py`、`core/shadow_ledger.py` | 领域单测、Python 语法、前端 build |
| 2. Overseer 协议升级 | 支持状态、计划、机制、触发、关系变化与空动作 | `core/undercurrent.py`、`system_prompts.yml` | 假 AI 行为测试与旧协议兼容 |
| 3. 世界推演实体 | 可载入、启用、封存的模板；不自动强制注入 | 实体模板、局内资产接入 | 新局与中途载入行为测试 |
| 4. 最小可见性 | 调试模式下按可见性等级展示实体信息 | API/UI 的实体展示层 | 前端 build 与手动接口测试 |

## v1.1.0 非目标

- 独立 `world_state` 表
- 全量 NPC 社会模拟或人格持久化
- 世界书自动演化
- `forecast` 预知玩法
- 关系图 UI、画图、对话/群聊、用户偏好分析
- 大规模实体的异步或多阶段推演

## 当前状态

切片 1 已开始：领域模型、仓储和账本已经从会话层拆出。完成全部验证和代码审阅后，才进入切片 2；不将旧 redesign 代码或 patch 直接合并到当前基准。
