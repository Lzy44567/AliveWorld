# AliveWorld 下一阶段设计总纲

> 代码基准：工作空间中的 `AliveWorld` 当前内容（2026-07-10 经用户确认）。
>
> 需求优先级：`功能想法_原始手写版本.txt` > 用户后续明确口述 > Codex 改版资料。

## 当前目标

下一阶段先实现“活的世界”：玩家视野之外的长期力量能够在世界规则约束下产生变化，并在合适的未来回合形成可追溯的后果。

活世界核心闭环是：世界书提供事实边界，暗流实体推动长期变化，暗流因果账本保存未兑现后果，动态未来候选在当前事实下选择近期走向，正文 AI 负责把选中结果写成玩家经历。对应的动态未来设计见 [`features/03_dynamic_futures.md`](features/03_dynamic_futures.md)。

世界书后续采用独立领域模块与可替换检索接口：预计算向量负责语义候选召回，关键词和系统标签负责强触发与降级，最终注入受上下文预算约束。世界书工坊作为独立聊天会话操作草稿，不与正文回合、实体或世界时间耦合。

这不是完整的软件重写。本阶段不扩展到完整 NPC 社会模拟、所有 NPC 的人格持久化、关系图 UI或群聊。`v1.5` 以“同回合只收集证据、低频异步分析、Python 贝叶斯更新、候选隔离和可撤销证据”为边界接入用户偏好学习；不会增加每回合独立分析线程。

`v1.1` 至 `v1.3` 已完成实体领域、因果账本、单层动态未来与世界书工具链的可运行 MVP。`v1.4.0-dev.1` 已建立系统型“世界推演”与普通暗流实体的职责协议；没有真实日志证据前不继续增加固定频率、强度或独立世界状态。

`v1.4.4-dev.1` 开始接入长篇故事分级记忆：完整原文永久保留，正文工作上下文改由 token 水位、完整回合前缀、章节脊柱、重点/核心记忆和低频故事事件共同组装。压缩模型不每回合运行；详细现状、实现常量、失败回退和可替换方案见 [`features/06_hierarchical_memory.md`](features/06_hierarchical_memory.md)。

重新规划后的 **v1.4.0 主开发线是生图基础版**：以本地 ComfyUI、手动角色立绘/剧情 CG、持久化异步任务、正文原位占位与本局画廊构成玩家可见的完整闭环。详细设计见 [`features/05_image_generation.md`](features/05_image_generation.md)，切片安排见 [`VERSION_ROADMAP.md`](VERSION_ROADMAP.md)。

## 对象边界

| 对象 | 职责 | 是否进入暗流推演 |
| --- | --- | --- |
| 世界书 | 世界规则、背景、常驻设定、触发条目 | 否；它是规则输入 |
| 暗流实体 | 组织、大 Boss、灾难、制度、世界级因果源 | 是 |
| 角色卡 | 可复用的人物资产 | 否 |
| NPC 状态 | 本局中人物的短期状态 | 默认否 |
| 状态栏 | 玩家可见数值与文本状态 | 否 |

普通路人、小怪、一次性人物不应自动成为暗流实体。只有具有长期动机、幕后行动能力、持续影响或世界级意义的对象才符合条件。

## MVP：暗流实体升级

1. 实体具备 `name`、`motive`、`status`、`recent_actions`、`plans`、`mechanisms`、`triggers`、`relationships`、`importance`、`is_active`。
2. 一个统筹级 Overseer 每回合读取世界书、近期剧情、活跃实体与影子账本；它可以返回空动作。
3. 默认系统型实体名为“世界推演”，不是“天道”或“世界意志”。后二者仅在具体世界观真实存在时作为故事内实体。
4. `is_active=false` 的实体保留在库中，但不参与下回合 Overseer 推演；MVP 不新增复杂全局开关系统。
5. 影子账本记录实体行动、计划、机制和触发线索，并控制注入长度。
6. 调试/玩法可见性预留 `hidden`、`names`、`motives`、`full`；`forecast` 属于 P3。

## 当前代码事实与改造原则

当前已经具备结构化实体模型与仓储、Overseer、暗流因果账本、动态未来、世界书检索/捕获/工坊及对应测试。当前缺口是：实体给 Overseer 的摘要过去没有明确类型和定位，导致系统型世界推演可能被当成普通角色或陷阱制造者。

改造继续采用功能驱动的局部重构：在现有实体摘要和 Overseer 协议上建立职责边界，不新增独立世界状态表，不为“世界推演”复制第二套推演引擎，也不把旧 redesign 工作树的代码直接套入本基准。

## 范围外与后置项

- 独立 `world_state` 大状态表
- 完整 NPC 社会模拟或全部 NPC 长期人格化
- 每回合自动重写或演化整本世界书
- 关系网或思维导图 UI
- 异步后台推演
- 12+ / 20+ 实体的多阶段筛选推演
- `forecast` 预知玩法
- 一次性展开多层 `N×n` 剧情树

## 预期演进

1. 已完成：实体领域边界、YAML 持久化、结构化影子账本与旧存档兼容。
2. 已完成：Overseer 状态、计划、机制、触发线索、关系变化与空动作协议。
3. 已完成：可按 `is_active` 启用或封存的世界推演模板，不强制新故事自动注入。
4. 已完成：实体名称、动机、编辑和气泡的可组合可见性。
5. 当前切片：把实体类型、标签和定位注入 Overseer，约束世界推演关注宏观变化、普通实体遵守自身能力与信息边界。
6. 后续需设计：世界规模判断、运行强度和调度策略；没有真实日志证据前不写死固定回合频率。
7. 当前主线：实现不阻塞正文的手动生图闭环；AI 自动判断生图和自动用户偏好学习后置。
8. v1.4.2 补齐基础游玩闭环：正文同次结算返回可点击编辑的行动建议；结算上下文与隐藏因果判断分离，避免建议泄露幕后信息。
9. v1.5 主线：渐进学习用户偏好；首版不要求玩家大量手填，也不把角色扮演行为直接认定为玩家偏好。模型负责提出多种心理解释和证据强度，Python 负责后验；遗漏原因通过保留原始证据、覆盖缺口和重新分析机制处理。详见 [`features/07_user_preferences.md`](features/07_user_preferences.md)。
10. v1.5.0-dev.4：偏好工坊以独立草稿和连续 Agent 对话整理偏好；AI 不能修改证据或后验，发布时在最新正式偏好卡上重放操作，避免覆盖游玩期间的新学习结果。
11. v1.5.0-dev.5：顶部总工坊统一世界书、角色、文风、实体和偏好入口。共同部分只负责资产选择、连续讨论、提案、草稿、撤销和发布；不同资产继续使用独立字段权限与提示词协议。详见 [`features/08_workshop_mode.md`](features/08_workshop_mode.md)。
12. v1.5.0-dev.6：工坊从弹窗启动器升级为顶层模式。游戏与工坊复用三栏视觉骨架但隔离状态：左侧草稿、中央协作、右侧资产；切换不关闭故事或丢失工坊草稿。

双轨中的“条件判断”不是新增模型阶段：现有变数推演读取完整暗流因果账本并返回条件是否满足；剧情结算只接收其中已经触发的影响并负责写成正文。撤回必须恢复整个暗流快照，包括账单状态、触发/尝试次数与因果回合计数。

每一切片都应独立验证，避免以一次大重构替代可回归的渐进开发。
