# AliveWorld 下一阶段设计总纲

> 代码基准：工作空间中的 `AliveWorld` 当前内容（2026-07-10 经用户确认）。
>
> 需求优先级：`功能想法_原始手写版本.txt` > 用户后续明确口述 > Codex 改版资料。

## 当前目标

下一阶段先实现“活的世界”：玩家视野之外的长期力量能够在世界规则约束下产生变化，并在合适的未来回合形成可追溯的后果。

活世界核心闭环是：世界书提供事实边界，暗流实体推动长期变化，暗流因果账本保存未兑现后果，动态未来候选在当前事实下选择近期走向，正文 AI 负责把选中结果写成玩家经历。对应的动态未来设计见 [`features/03_dynamic_futures.md`](features/03_dynamic_futures.md)。

世界书后续采用独立领域模块与可替换检索接口：预计算向量负责语义候选召回，关键词和系统标签负责强触发与降级，最终注入受上下文预算约束。世界书工坊作为独立聊天会话操作草稿，不与正文回合、实体或世界时间耦合。

这不是完整的软件重写。本阶段不扩展到完整 NPC 社会模拟、所有 NPC 的人格持久化、世界书自动演化、关系图 UI、画图、群聊或用户偏好分析。

本阶段的版本目标是 **v1.1.0：活的世界 MVP**。具体迭代顺序、验收条件和版本管理见 [`VERSION_ROADMAP.md`](VERSION_ROADMAP.md)。

## 对象边界

| 对象 | 职责 | 是否进入暗流推演 |
| --- | --- | --- |
| 世界书 | 世界规则、背景、常驻设定、触发条目 | 否；它是规则输入 |
| 暗流实体 | 组织、大 Boss、灾难、制度、世界级因果源 | 是 |
| 角色卡 | 可复用的人物资产 | 否 |
| NPC 状态 | 本局中人物的短期状态 | 默认否 |
| 状态栏 | 玩家可见数值与文本状态 | 否 |

普通路人、小怪、一次性人物不应自动成为暗流实体。只有具有长期动机、幕后行动能力、持续影响或世界级意义的对象才符合条件。

## MVP：暗流实体升级

1. 实体具备 `name`、`motive`、`status`、`recent_actions`、`plans`、`mechanisms`、`triggers`、`relationships`、`importance`、`is_active`。
2. 一个统筹级 Overseer 每回合读取世界书、近期剧情、活跃实体与影子账本；它可以返回空动作。
3. 默认系统型实体名为“世界推演”，不是“天道”或“世界意志”。后二者仅在具体世界观真实存在时作为故事内实体。
4. `is_active=false` 的实体保留在库中，但不参与下回合 Overseer 推演；MVP 不新增复杂全局开关系统。
5. 影子账本记录实体行动、计划、机制和触发线索，并控制注入长度。
6. 调试/玩法可见性预留 `hidden`、`names`、`motives`、`full`；`forecast` 属于 P3。

## 当前代码事实与改造原则

当前 `core/undercurrent.py` 已有实体、Overseer 与影子账本雏形，但实体只有 `name + goal`，且实体文件同步逻辑位于 `core/game_session.py`。这不足以承载 MVP。

改造采用功能驱动的局部重构：为实体领域建立责任单一的模型、持久化、账本和 Overseer 解析边界，再通过一个窄接口接入会话流程。不要先进行全项目“大爆炸式”重构，也不要把旧 redesign 工作树的代码直接套入本基准。

## 范围外与后置项

- 独立 `world_state` 大状态表
- 完整 NPC 社会模拟或全部 NPC 长期人格化
- 世界书 AI 自动扩写/自动演化
- 关系网或思维导图 UI
- 异步后台推演
- 12+ / 20+ 实体的多阶段筛选推演
- `forecast` 预知玩法
- 一次性展开多层 `N×n` 剧情树

## 预期演进

1. 实体领域边界：模型、YAML 持久化、结构化影子账本、旧存档兼容；不改变 UI 与提示词协议。
2. Overseer 协议：让统筹输出实体状态、计划、机制、触发线索和关系变化；允许空动作。
3. 世界推演：提供可按 `is_active` 启用或封存的实体模板，不强制新建故事自动注入。
4. 可见性：沿用现有调试入口，以最小 UI 展示 `hidden / names / motives / full`。

每一切片都应独立验证，避免以一次大重构替代可回归的渐进开发。
